Whats New

New features

Workload Tab

New Workload tab is introduced in this build. Workload tab encompases all necessary tools to enable users to build their own application on their instance of vitalpointz IoT Core Platform, deploy them securely, monitor and manage them throughout the life of the application.

Workload tab is divided into following main areas  - Template Ribbon, Container Management area and Registry.

 

Template ribbon

This shows the available container images. The images are hosted under the vitalpointz managed IoT Service platform aka vMIST. The list is updated periodically as and when new images are supported.

By clicking on the card in the template ribbon, a new container of the desired type can be spun up.

Container can be mounted with a storage volume. This is typically useful for database containers or any container that requires persisitent storage.

 

'Always On' check box allows the container to be exposed to external world using URL provided by the user.

If 'Always On' is unchecked, the container's ports are not exposed to external world but remains available for other containers.

Container Management area

 Once the container is deployed, it is represented as a card in this area. Each card representing a container has bunch of controls based on the current state of the container.

 

Clicking on the card shows a slider which shows additional information about the container.

 

 

Registries

 There are two kinds of registries listed under the workload tab namely Public and Private. Public registry service runs as part of vMIST (vitalpointz Management IoT Service) services. Private Registry service runs locally as part of every instance of IoT Core.

 

Typical workflow with containers in vitalpointz IoT Core

Public Regsitry hosts the images tested, modifed with debugging abilities. User can spin up a container from the public registry, modify if necessary, commit the modified container as image into Private Registry. Then spin up production containers from Private Registry.

  

Outport Tab Improvements

 Outport Module is updated with a logstash feeder service. This service can be invoked using Outport framework. Adding a new outport instance shows option to use REST service or Logstash feeder service. 

 When LOGSTASH is selected, index name is prompted from the user. This is useful when ELK Stack is instantiated using workload manager described earlier. 

 

 

New TCP Streaming Interface added

This version of vitalpointz IoT Core supports TCP Streaming IoT devices. The Service is available on a specific URL and port numnber as other MQTT services. This can be viewed under Device Tab, clicking on top most tab. The slider shows the list of interface services. 

 

 

Adding TCP Streaming Devices

In order to add a TCP Streaming device, click on Device Tab, Add IoT device under the slider of the Top most card. 

Click on 'Auto' and click "upload file" option. A text file with list of device identity (such as IMSI number) has to be uploaded. one Entry per line and the identity can be any non space, non special character containing string of alpha-numneric identity. 

 This adds the devices - one per unique identity - and provisions the TCP interface service. 

 Device is expected to send its data in a CSV form with its own unique identity is sent at the 2nd field. When device connects to TCP Service and sends data, the TCP interface service receives them , parses them and using the unique id that is pre-uploaded, the data is published in using its allocated topic.

In other words, the device data is available to workload or node red tool as if it came from regular MQTT channel with its own Topic.
